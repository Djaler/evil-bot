# Дизайн: детектор дубликатов медиа (фото, видео, GIF)

Дата: 2026-06-12
Статус: на ревью

## Контекст и проблема

Текущая фича «Уже было» (`SeenMemeHandler` + `DuplicateImageChecker`) ловит дубликаты картинок ненадёжно:

- Хеш — MD5 от PNG-байтов картинки, уменьшенной до 64×64. Telegram пережимает каждую загрузку, поэтому та же картинка от другого пользователя даёт другие пиксели → другой MD5 → дубликат не ловится. Фактически ловятся только форварды того же файла.
- `BufferedImage(w, h, image.type)` бросает `IllegalArgumentException` при `image.type == TYPE_CUSTOM` (например, CMYK-JPEG) — такие фото не обрабатываются вовсе.
- Скачивается полноразмерное фото, хотя для хеша хватает мини-превью.
- Видео и GIF не покрыты.

## Принятые решения

| Вопрос | Решение |
|---|---|
| Уровень детекции | «Визуально та же картинка»: перцептивный хеш (dHash), устойчивый к пережатию/ресайзу. Кропы, повороты, перебитый текст — вне скоупа. |
| Алгоритм | Свой dHash, 64 бита, без новых зависимостей |
| Поиск | Расстояние Хэмминга ≤ 5 через `bit_count(hash # :hash)` в Postgres (15-alpine, обновление не требуется) |
| Индекс | Линейный скан внутри чата (btree по `chat_id`). Pigeonhole-индекс — отложен, путь миграции аддитивен (см. «Масштабирование») |
| Типы медиа | Фото (`PhotoContent`), видео (`VideoContent`), GIF (`AnimationContent`). Видеосообщения-кружочки — вне скоупа |
| Старые данные | Чистый лист: создаётся новая таблица `media_hashes`; старая `image_hashes` остаётся нетронутой и удаляется отдельной миграцией после проверки фичи |

## Архитектура

Поток не меняется: `UpdatesManager` → `SeenMemeHandler` → `DuplicateMediaChecker` → репозиторий.

### 1. Схема БД и миграция

Flyway-миграция `V10` создаёт новую таблицу (старая `image_hashes` не трогается):

```sql
CREATE TABLE media_hashes
(
    id         BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    hash       BIGINT   NOT NULL,
    chat_id    SMALLINT NOT NULL REFERENCES chats (id) ON DELETE CASCADE,
    message_id BIGINT   NOT NULL,
    file_id    TEXT     NOT NULL
);

CREATE INDEX media_hashes_chat_id_idx ON media_hashes (chat_id);
```

- без `UNIQUE (chat_id, hash)` — при поиске «по близости» точная уникальность не инвариант;
- `file_id` сохраняем (отладка, потенциальный бэкфилл);
- старая таблица `image_hashes` удаляется отдельной миграцией после проверки фичи в проде.

Новые сущность `MediaHash` (`hash: Long`) и репозиторий `MediaHashRepository` с нативным запросом; старые `ImageHash`/`ImageHashRepository` после перехода кода не используются и удаляются сразу (таблицу это не затрагивает):

```sql
SELECT * FROM media_hashes
WHERE chat_id = :chatId AND bit_count(hash # :hash) <= :maxDistance
ORDER BY id
LIMIT 1
```

`ORDER BY id` — при накоплении нескольких близких хешей (гонка двух одновременных постов) ссылка ведёт на самый ранний «оригинал», а не на произвольную совпавшую строку.

### 2. Сервис хеширования

`DuplicateImageChecker` → `DuplicateMediaChecker`:

- `dHash(image: BufferedImage): Long` — internal top-level функция (тестируется без моков репозитория; класс сервиса живёт в том же файле):
  1. уменьшение до 9×8 через `Image.SCALE_SMOOTH` (area-averaging; устойчивее однопроходной билинейной интерполяции на такой маленькой цели), рисование на канву фиксированного типа `TYPE_INT_RGB` — устраняет баг с `TYPE_CUSTOM`; результат `drawImage` проверяется через `check` — страховка от молчаливого нулевого хеша при несинхронном рендере;
  2. перевод в оттенки серого;
  3. в каждой из 8 строк — 8 сравнений соседних пикселей «левый ярче правого» → 64 бита → `Long`.
- `findDuplicate(image, chat): Long?` — dHash + запрос с порогом `MAX_HAMMING_DISTANCE = 5` (константа в сервисе);
- `saveHash(image, chat, messageId, fileId)` — как сейчас, но с `Long`-хешем.

### 3. Хендлер

`SeenMemeHandler` обобщается: из сообщения извлекается `PhotoSize` для хеширования:

- `PhotoContent` → самый маленький элемент `mediaCollection` (для 9×8 хватает, минимальный трафик вместо текущего полноразмерного скачивания);
- `VideoContent` → `media.thumbnail`;
- `AnimationContent` → `media.thumbnail`;
- thumbnail отсутствует (`null`) → сообщение пропускается (`return false`).

Дальше единый путь: скачать `PhotoSize` → `ImageIO.read` → `findDuplicate` → ответ «Уже было + ссылка на оригинал» / `saveHash`. Поведение в чате (текст ответа, остановка цепочки хендлеров) не меняется.

### 4. Обработка ошибок

- `ImageIO.read` вернул `null` или бросил (битый файл) → пропустить сообщение (`return false`), цепочка хендлеров не прерывается;
- ошибка скачивания из Telegram → аналогично;
- существующий `ExceptionsManager` остаётся последней линией обороны.

### 5. Тесты (первые в проекте)

Юнит-тесты `dHash` на синтетических картинках (генерируются кодом с фиксированным seed, бинарных фикстур нет):

- та же картинка после JPEG-пережатия (quality 0.3) и после ресайза → расстояние ≤ 5;
- две разные картинки → расстояние > 10;
- однотонная картинка хешируется без ошибок;
- картинка с `TYPE_CUSTOM` (синтетическая, 16-битный `ComponentColorModel`) обрабатывается без исключений. Реальный CMYK-JPEG фикстурой не покрыт: фикс типонезависим (канва всегда `TYPE_INT_RGB`), а свойство `TYPE_CUSTOM` тест утверждает явно.

Корректность нативного запроса с `bit_count` проверяется на живой базе при ручной проверке (Testcontainers в проект не добавляем).

## Масштабирование (отложено, путь зафиксирован)

На текущем масштабе (тысячи–десятки тысяч строк на чат, единицы запросов в минуту) линейный скан — единицы миллисекунд на фоне 100–500 мс скачивания превью из Telegram.

Если чат накопит сотни тысяч строк, точный индексированный поиск добавляется чисто аддитивно (pigeonhole-декомпозиция, консультация sql-pro):

- 6 колонок `SMALLINT GENERATED ALWAYS AS ((hash >> N) & mask) STORED` (4×11 бит + 2×10 бит);
- 6 btree-индексов `(chat_id, chunk_i)`;
- запрос: `OR` из 6 равенств (BitmapOr) + контрольный `bit_count <= 5` на кандидатах — результат точный;
- ничего не пересчитывается: колонки вычислятся из накопленных хешей при миграции.

## Отклонённые альтернативы

- **JImageHash (pHash)** — та же точность на нашем сценарии, но новая малоактивная зависимость и неудобное хранение (`BigInteger` → `bytea` либо усечение до 64 бит, обессмысливающее библиотеку).
- **dHash с точным совпадением** — пережатие сдвигает 0–4 бита из 64, заметная доля дублей пропускалась бы непредсказуемо.
- **pgvector (HNSW по `bit(64)`)** — индекс приближённый (recall 95–99%), встроенные пропуски дублей противоречат сути фичи.
- **bktree (GiST)** — точный индекс, но заброшенный сторонний C-extension с компиляцией в кастомный Docker-образ.
