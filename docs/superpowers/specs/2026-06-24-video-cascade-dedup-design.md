# Дизайн: каскадное определение дубликатов видео

Дата: 2026-06-24
Статус: на ревью

## Контекст и проблема

Текущая детекция дубликатов (`SeenMemeHandler` + `DuplicateMediaChecker`, введена в 3d0a161)
для видео считает перцептивный dHash по **одному кадру** — Telegram-овскому `thumbnail`.
Часто это первый кадр ролика, и у разных видео с чёрной заставкой превью совпадает →
одинаковый dHash → ложное «уже было». Хеш по обложке принципиально не различает контент
за одинаковой обложкой.

Жалоба пользователя: попались разные видео с одинаковым чёрным превью, бот счёл их дубликатами.

Что уже есть и **не меняется**:
- Фото (`PhotoContent`) — детекция по dHash самого маленького превью. Корректна, не трогаем.
- GIF (`AnimationContent`) — поддержка удалена в #235 (a8f9a21), повторы гифок это норма. Не возвращаем.
- TTL/`last_seen_at`, `MediaHashCleanupScheduler`, hamming ≤ 5, нативный `bit_count`-поиск.

Скоуп этой работы — **только видео** (`VideoContent`).

## Идея: каскад «дёшево → дорого»

Дешёвые сигналы из метаданных (без скачивания) отсеивают почти все не-дубликаты;
скачивание видео + извлечение кадров через ffmpeg включается **только** для реально
неоднозначных кандидатов (совпали и обложка, и длительность).

```
видео (VideoContent)
  └─ Tier 0 (метаданные, без сети): dHash(thumbnail) + duration
       ├─ нет кандидата (обложка ИЛИ длительность отличаются) → запись, молчим   ← типичный путь, не качаем
       └─ есть кандидат (обложка И длительность совпали) → Tier 1
            └─ Tier 1 (скачать видео + ffmpeg, кэш кадров кандидата лениво):
                 ├─ кадры совпали            → «уже было» + ссылка на оригинал
                 ├─ кадры не совпали          → запись, молчим
                 └─ проверка невозможна (>20МБ / ошибка) → запись, молчим
фото → как сейчас, без изменений
```

## Принятые решения

| Вопрос | Решение |
|---|---|
| Tier 0 — дешёвый шлюз | dHash(thumbnail) **И** совпадение `duration` (±1 с). Оба должны совпасть, чтобы эскалировать |
| `duration` источник | `VideoFile.duration: Long?` (секунды), из метаданных. Если `null` — гейт по длительности пропускаем, эскалируем по одному хешу |
| Tier 1 — кадры | ffmpeg достаёт 4 кадра на относительных позициях **20/40/60/80 %** (смещены от краёв — обходят чёрные интро/аутро); dHash каждого → вектор `List<Long>` |
| Сравнение векторов | покадрово (по индексу) hamming ≤ 5; дубликат, если совпало **большинство** кадров (≥ 3 из 4; при разной длине — большинство от min длины) |
| Ленивое кэширование | вектор кадров кандидата сохраняется в БД при первой эскалации; повторные коллизии не качают кандидата заново |
| Лимит размера | Bot API `getFile` не отдаёт файлы >20 МБ. Если `VideoFile.fileSize` > 20 МБ — Tier 1 не выполняем |
| Фолбэк при невозможности Tier 1 | **молчим** (считаем не-дубликатом). Приоритет — ноль ложных срабатываний (суть жалобы). Реальный дубль крупного видео изредка пропустим |
| Транзакции | скачивание + ffmpeg (секунды) — вне транзакции. БД-операции — короткие отдельные транзакции |

## Архитектура

Поток хендлеров не меняется: `UpdatesManager` → `SeenMemeHandler`.
Внутри `SeenMemeHandler.handleMessage` ветвление по типу контента уже есть; в нём:

- **фото** → прежний путь без изменений: `DuplicateMediaChecker.checkAndRecord(...)`;
- **видео** → новый каскад (оркестрация в хендлере: у него уже есть `content`, `chat`,
  `requestExecutor`).

### 1. Схема БД и миграция

Новая Flyway-миграция `V12__add_media_hashes_video_columns.sql` — аддитивно:

```sql
ALTER TABLE media_hashes ADD COLUMN duration    BIGINT;
ALTER TABLE media_hashes ADD COLUMN frame_hashes BIGINT[];
```

- оба nullable; для фото остаются `NULL`;
- старые видеозаписи (`duration IS NULL`) просто не совпадут по длительности с новыми видео
  на Tier 0 → будут трактоваться как новые и вытеснятся по TTL — безопасно;
- по `frame_hashes` в SQL **не ищем** — это кэш для сравнения в коде на конкретных кандидатах.

Сущность `MediaHash` дополняется полями `duration: Long?`, `frameHashes: List<Long>?`.
`frame_hashes BIGINT[]` маппится через Hibernate 6 `@JdbcTypeCode(SqlTypes.ARRAY)` на
`List<Long>?` (Spring Boot 3 / Hibernate 6 поддерживает массивы нативно). Если маппинг
массива окажется проблемным — фолбэк на дочернюю таблицу `media_hash_frames(media_hash_id,
position, hash)`; решение принимается на этапе реализации, на дизайн не влияет.

### 2. Репозиторий

`MediaHashRepository` — текущий метод `findByChatIdAndHashCloseTo` остаётся (фото и
видео-без-длительности). Добавляется тиро-0-запрос с гейтом по длительности:

```sql
SELECT * FROM media_hashes
WHERE chat_id = :chatId
  AND bit_count((hash # :hash)::bit(64)) <= :maxDistance
  AND duration BETWEEN :duration - 1 AND :duration + 1
ORDER BY id
```

- возвращает **список** кандидатов (не `LIMIT 1`): один thumbnail+duration может совпасть
  с несколькими записями; Tier 1 сверяет контент по каждому, пока не найдёт совпадение;
- `ORDER BY id` — оригиналом считается самая ранняя запись.

Методы для записи/обновления (короткие транзакции, вызываются из хендлера — proxy работает):
- сохранить новую запись видео (`hash`, `chatId`, `messageId`, `fileId`, `lastSeenAt`,
  `duration`, `frameHashes = null` на Tier 0);
- сдвинуть существующую запись на текущее сообщение (как сейчас: `messageId` + `lastSeenAt`);
- до-сохранить `frameHashes` кандидату (ленивый кэш).

### 3. Сервис извлечения кадров — `VideoFingerprintService`

Новый `@Service` (не транзакционный), зависит от `RequestsExecutor`:

- `suspend fun extractFrameHashes(file: VideoFile): List<Long>?`
  1. если `file.fileSize != null && file.fileSize > 20 МБ` → вернуть `null` (не качаем);
  2. `requestExecutor.downloadFile(file.fileId)` → временный файл (`Files.createTempFile`);
  3. позиции кадров: из `file.duration` берём 20/40/60/80 %; если `duration == null` —
     фиксированные секунды-фолбэк (напр. 1/2/3/4 с), лишние/несуществующие кадры просто не
     извлекутся;
  4. на каждую позицию — ffmpeg (паттерн из `VideoConvertService`, ProcessBuilder,
     блокирующе → `withContext(Dispatchers.IO)`):
     `ffmpeg -hide_banner -loglevel error -ss <sec> -i <input> -frames:v 1 -y <out.png>`;
  5. `ImageIO.read` каждого PNG → `dHash(...)` (переиспользуем существующую top-level `dHash`);
  6. временные файлы удаляются в `finally`;
  7. при ошибке скачивания/ffmpeg/декодирования или если извлеклось < 2 кадров → `null`
     (Sentry.captureException, далее каскад трактует как «проверка невозможна»).

### 4. Оркестрация в `SeenMemeHandler` (ветка видео)

1. `thumbnail = content.media.thumbnail ?: return false` (нет превью — пропускаем);
2. скачать thumbnail → `BufferedImage` → `thumbHash = dHash(image)` (как сейчас);
3. `duration = content.media.duration`;
4. **Tier 0**: запросить кандидатов
   (`duration != null` → запрос с гейтом по длительности; `duration == null` → прежний
   hash-only запрос). Нет кандидатов → сохранить новую запись (с `duration`,
   `frameHashes = null`) → `return false` (**не качаем**);
5. **Tier 1** (есть кандидаты):
   - `newFrames = videoFingerprintService.extractFrameHashes(content.media)`;
   - `newFrames == null` (>20МБ/ошибка) → сохранить новую запись → `return false` (молчим);
   - для каждого кандидата по порядку (`ORDER BY id`):
     - `candFrames = candidate.frameHashes` ?: `extractFrameHashes(candidate.fileId-as-VideoFile)`
       — кандидат хранит только `fileId`, поэтому для него извлекаем кадры по `fileId`
       (см. ниже про размер) и **до-сохраняем** в БД (ленивый кэш). Если извлечь не удалось —
       пропускаем этого кандидата;
     - совпадение векторов (большинство покадрово) → ответить «уже было» + ссылка на
       оригинал, сдвинуть запись кандидата на текущее сообщение → `return true`;
   - ни один кандидат не совпал по контенту → сохранить новую запись → `return false` (молчим).

Замечание по кандидату: в БД хранится `fileId` (строка), а `extractFrameHashes` принимает
`VideoFile`. Для до-вычисления кадров кандидата достаточно `fileId` и (для лимита 20 МБ)
`fileSize`. Вариант реализации: добавить перегрузку `extractFrameHashes(fileId: FileId,
fileSize: Long?, duration: Long?)`, а версию для `VideoFile` сделать тонкой обёрткой. Размер
кандидата на момент сохранения не фиксировали — для кандидата лимит проверяем по факту
(ошибка скачивания >20МБ → `null` → кандидат пропускается). Деталь реализации, на дизайн не влияет.

### 5. Конкретные параметры (тюнятся позже, в коде — именованные константы)

- `MAX_HAMMING_DISTANCE = 5` (уже есть, переиспользуем для покадрового сравнения);
- `FRAME_POSITIONS = [0.2, 0.4, 0.6, 0.8]`;
- `MIN_MATCHING_FRAMES`: большинство = `ceil(размер_сравнения / 2)` (для 4 кадров → 3);
- `MAX_DOWNLOAD_BYTES = 20 * 1024 * 1024`;
- `DURATION_TOLERANCE_SECONDS = 1`.

### 6. Обработка ошибок

- thumbnail отсутствует / битый → `return false` (как сейчас);
- любая ошибка Tier 1 (скачивание, ffmpeg, декодирование) → логируем + Sentry,
  каскад трактует как «проверка невозможна» → молчим;
- `ExceptionsManager` остаётся последней линией обороны.

### 7. Транзакции и конкурентность

- Tier 0-запрос, сохранение новой записи, сдвиг записи, до-сохранение кадров — каждая
  отдельная короткая транзакция; скачивание + ffmpeg между ними — **вне** транзакции;
- редкая гонка (два одинаковых видео одновременно) может пропустить дубль — допустимо,
  как и сейчас; `ORDER BY id` сохраняет ссылку на ранний оригинал.

## Тесты

В стиле существующего `DHashTest` (без Spring-контекста, синтетика):

- сравнение векторов кадров: одинаковые векторы → дубликат; векторы, различающиеся в 2+
  кадрах → не дубликат; разная длина векторов → сравнение по min длины;
- логика «большинство ≥ 3/4»;
- (если выделить чистую функцию) выбор позиций кадров из `duration`, фолбэк при `null`.

ffmpeg-извлечение и нативный SQL с гейтом по длительности проверяются вручную на живой базе
(Testcontainers/ffmpeg-в-тестах в проект не добавляем — консистентно с текущим решением).

## Отклонённые альтернативы

- **Только длительность, без кадров** — не различает разные видео с одинаковой обложкой
  И длительностью; не закрывает суть бага.
- **Всегда качать видео и хешировать кадры** (без Tier 0) — ffmpeg на каждое сообщение,
  лишний трафик, лимит 20 МБ бьёт по всем. Каскад делает дорогую проверку редкой.
- **Кросс-матч фото↔видео по обложке** — вне скоупа (жалоба про видео↔видео); фото и видео
  с разными `duration` (у фото `NULL`) и так не совпадут на Tier 0.
- **Фолбэк «считать дубликатом» при >20МБ** — оставляет ровно то ложное срабатывание,
  на которое жалоба. Выбрано «молчать».
- **ffprobe для длительности при `duration == null`** — лишняя зависимость в пути; хватает
  фиксированного фолбэк-набора секунд с толерантностью к отсутствующим кадрам.
